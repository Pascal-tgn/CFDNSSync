CfDnsSync

One-way DNS synchronization from Cloudflare to a local Active Directory Domain Controller. Keeps your internal DNS zone in sync with Cloudflare — automatically, safely, without touching your AD.

Overview

CfDnsSync is a Windows Service that periodically fetches DNS records from your Cloudflare zone and replicates them into a local AD-integrated DNS zone on your Domain Controller. Synchronization is strictly one-way: Cloudflare → DC. The service never modifies Cloudflare.

⚡ One-way sync

Cloudflare is the source of truth. The DC zone is kept in sync. Never writes back to Cloudflare.

🔒 Safe by default

Starts in Dry Run mode. No changes are applied until you explicitly enable live sync.

🎛️ Ownership model

Every record is tagged as CF Managed, DC Managed, or Conflict. DC-only records are never touched.

🌐 Web dashboard

HTTPS dashboard with Windows authentication. View records, resolve conflicts, adjust settings.

Requirements

  • Windows Server 2016 or later (Domain Controller)
  • .NET 8 runtime — or use the self-contained build (no runtime needed)
  • Cloudflare account with API token access (Zone → DNS → Read permission)
  • Local Administrator rights on the DC for installation
  • DNS Server role installed on the DC

Installation

Step 1 — Build

On a Windows machine with .NET 8 SDK:

dotnet publish src\CfDnsSync.csproj -c Release -r win-x64 --self-contained true ^
  -p:PublishSingleFile=true -p:EnableCompressionInSingleFile=true -o .\publish

Step 2 — Copy to DC

Copy publish\CfDnsSync.exe and deploy\Install-CfDnsSync.ps1 to the Domain Controller (e.g. C:\Temp\).

Step 3 — Run installer

# Run as Administrator in PowerShell
.\Install-CfDnsSync.ps1 -StartAfterInstall

# Custom install directory and port
.\Install-CfDnsSync.ps1 -InstallDir "D:\Services\CfDnsSync" -Port 9000 -StartAfterInstall

The installer will:

  1. Copy the binary to the install directory
  2. Register the Windows Service (LocalSystem, auto-start)
  3. Create a custom Windows Event Log source
  4. Run interactive configuration (Zone ID, domain, etc.)
  5. Encrypt and store your Cloudflare API token
  6. Create a Windows Firewall Allow rule for the dashboard port
  7. Configure service recovery (auto-restart on failure)
Token alternative: Instead of interactive setup, you can place your Cloudflare API token in a file named token.txt in the install directory. On next service start, the token is automatically encrypted with DPAPI and the plaintext file is deleted.

First Run

After installation, the service starts in Dry Run mode — it simulates all sync operations without making any changes to DC DNS. This lets you review planned changes before going live.

1

Open the dashboard

Navigate to https://<dc-hostname>:8765 in your browser. Accept the self-signed certificate warning on first launch.

2

Review planned changes

The Dashboard tab shows WOULD ADD / WOULD UPDATE / WOULD DELETE entries — what would happen if live sync were enabled.

3

Resolve conflicts

Go to the Records tab. Records shown as Conflict have different values on CF and DC. Assign ownership manually.

4

Enable live sync

Go to Settings, uncheck Dry Run Mode, and save. A confirmation dialog will appear. After saving, the next sync cycle will apply real changes.

Dry Run Mode

Dry Run mode is enabled by default. In this mode the service runs the full reconciliation logic — fetching records from Cloudflare, comparing with the DC zone, and computing all required changes — but does not apply any changes to DNS.

The Dashboard shows planned changes as WOULD ADD, WOULD UPDATE, and WOULD DELETE. A yellow banner indicates dry run is active.

To enable live sync: go to Settings → Dry Run Mode, uncheck the checkbox, and save. A confirmation dialog prevents accidental activation.

Note: If you redeploy or reset config.json, the service returns to Dry Run mode automatically.

Skipped Records

Some DNS records cannot be reliably synced due to limitations of the Windows DNS PowerShell module. TXT records on nodes with dots in the name (DKIM selectors like selector1._domainkey) are added successfully via dnscmd but cannot be read back by Get-DnsServerResourceRecord in a zone-wide enumeration, causing the service to re-add them on every cycle.

Add such records to Settings → Skipped Records (one per line). The service will not attempt to sync them at all. Manage them manually on the DC.

selector1._domainkey
selector2._domainkey

Record Ownership

Every DNS record in the zone is assigned an ownership mode:

ModeBehaviour
CF ManagedOwned by Cloudflare. Service keeps it in sync: creates, updates, and eventually deletes it from DC if it disappears from CF.
DC ManagedOwned by the DC. Service never touches it — no updates, no deletions. Use this for internal-only records.
ConflictExists on both sides with different type or value. Service skips it until you resolve ownership in the Records tab.

Auto-detection logic

  • Record only in Cloudflare → automatically CF Managed
  • Record only on DC → automatically DC Managed
  • Record on both sides, same type and same value → automatically CF Managed
  • Record on both sides, different type or value → Conflict, requires manual resolution

To change ownership: go to the Records tab, find the record, and click the CF or DC toggle.

Orphan Records

When a CF-managed record disappears from Cloudflare (e.g. it was deleted), the service does not delete it from the DC immediately. Instead it marks it as an Orphan and increments a cycle counter on every sync.

Once the counter reaches orphanDeleteAfterCycles (default: 3), the record is deleted from the DC automatically. This grace period protects against accidental deletions due to temporary Cloudflare API issues.

Safety: If Cloudflare returns 0 records (API error, expired token, network issue), the sync is aborted and orphan deletion is skipped — preventing accidental wipe of all CF-managed records.

Conflicts

A conflict occurs when a record exists on both CF and the DC but with different types or values. Common examples:

  • A record exists as an A record on the DC but as a CNAME in Cloudflare (e.g. a hostname that was converted to a CNAME redirect)
  • A TXT record with different content on each side

To resolve: go to the Records tab, find the conflicted record, and click CF (let Cloudflare version win) or DC (keep the DC version and ignore Cloudflare).

The web dashboard is accessible at https://<dc-hostname>:8765. It requires Windows authentication — you must be a member of the configured AD group (default: Domain Admins).

SSL Certificate

On first start, a self-signed certificate is auto-generated and stored in LocalMachine\My. The certificate thumbprint and netsh binding are configured automatically.

To use a trusted certificate, set certificateThumbprint in config.json to the thumbprint of an existing cert in LocalMachine\My.

Logging

CfDnsSync uses Serilog with two sinks:

  • Rolling file logs — all levels (Debug and above), stored in <install-dir>\logs\cfds-YYYYMMDD.log, retained for 30 days (configurable via maxLogAgeDays)
  • Windows Event Log (Application source: CfDnsSync) — errors only, plus SyncWorker warnings (startup/token failures)

Operational noise (PowerShell batch errors, CF retry warnings, etc.) goes to file only and does not appear in Event Viewer.

CLI Commands

# Interactive first-time configuration
CfDnsSync.exe setup-config

# Encrypt and store Cloudflare API token
CfDnsSync.exe setup-token

# Run one sync cycle immediately (for debugging)
CfDnsSync.exe sync-now
Important: setup-token must be run as the same Windows account that runs the service (LocalSystem by default). The token is encrypted with DPAPI CurrentUser scope and can only be decrypted by that account.

Uninstalling

# Run as Administrator
.\Uninstall-CfDnsSync.ps1

# Keep config and data files
.\Uninstall-CfDnsSync.ps1 -KeepData

The uninstall script stops the service, removes it from the Service Control Manager, removes the Firewall rule, removes the custom Event Log source, and optionally deletes all data files.

Troubleshooting

Service fails to start — "Key not valid for use in specified state"

The DPAPI token was encrypted under a different Windows account. Re-run setup-token as the service account, or place a token.txt file in the install directory and restart the service.

Records keep appearing as [ADDED] every sync cycle

The record exists on the DC but the service cannot read it back (common with dotted-name TXT records like DKIM selectors). Add the record name to Settings → Skipped Records.

Dashboard shows "DRY RUN MODE" banner

The service is in simulation mode. Go to Settings, uncheck Dry Run Mode, and save to enable real sync.

record_modes.json is very large

Old DC records (e.g. from decommissioned machines) accumulate over time. They are automatically purged after recordModesRetentionDays (default 90). You can also delete the file to reset all ownership assignments — the service will rebuild it from scratch on the next sync.

WIN32 9711 errors in logs

These are "record already exists" errors, silently treated as skipped. They appear in the log file but not as warnings. No action needed.

HRESULT 0x800706be errors

RPC failure when adding very long TXT records (e.g. DKIM keys). The service automatically falls back to dnscmd.exe for TXT records. If this persists, add the record to Skipped Records.

Security Notes

  • DPAPI CurrentUser scope: The Cloudflare API token is encrypted with Windows DPAPI under the service account. Only that account can decrypt it.
  • NTLM authentication: The web dashboard uses Windows NTLM authentication. Only members of the configured AD group can access it.
  • Self-signed TLS: The dashboard uses TLS by default. Replace the self-signed cert with a CA-issued cert for production use.
  • Read-only Cloudflare token: Use a Cloudflare API token with Zone → DNS → Read permission only. The service never writes to Cloudflare.
  • Dry Run by default: No DNS changes are applied until you explicitly disable Dry Run mode.
